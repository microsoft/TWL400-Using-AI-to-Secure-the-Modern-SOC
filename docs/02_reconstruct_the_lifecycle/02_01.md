# Task 02.01 — Hunt the unified Sentinel telemetry (KQL) and place each stage

## Introduction

The Exercise 01 narrative told you *what* happened; now you prove it from the data. Microsoft Sentinel is Zava's unified data backbone — the commodity stages (identity, email/poisoning, infrastructure, endpoint) and the ~25 decoy noise events are injected into the custom `SocLabEvents_CL` table, while the AI-layer attack raised a **real Defender for Cloud — AI threat protection alert** that is queryable through `SecurityAlert` / `AlertEvidence` in Advanced Hunting. Hunting these tables directly does two things alert-driven triage can't: it puts every stage on one timeline at the same fidelity, and it surfaces the AI-layer alert that fired but was never correlated into the cross-layer incident — buried among the decoys. By the end of this task each of the five attack stages will be placed on the lifecycle and linked back to the entities you anchored in Exercise 01.

## Description

You will run a series of KQL hunting queries in Microsoft Sentinel — one per attack stage — against the Sentinel-backed and custom-seeded tables. For each stage you will confirm the event, pull the timestamp and the entity, and record where it sits on the lifecycle. You will pivot on the two lead entities from Exercise 01 — **Mariya Petrova** (identity) and **`sp-refund-agent-inference`** (infrastructure) — and stitch the results into a single, evidence-backed sequence.

## Success Criteria

- You have run a hunting query per stage and confirmed each event in telemetry, capturing its timestamp and primary entity
- **All five stages placed and linked end to end** _(Validation: all five stages placed and linked end to end)_
- Each placed stage references an entity carried forward from Exercise 01 (Mariya Petrova, `billing@zava-refunds[.]net`, `102.89.42.17`, the Refund Policy Knowledge Base / `Vendor-Refund-Policy-Update-Q3.docx`, the Zava Refund Agent, `sp-refund-agent-inference`, OPS-LT-0427)
- You can name which stage's alert fired but was **left uncorrelated and buried in the noise** (the AI layer) — the gap you'll formalize in Task 02.03

## Key Tasks

{: .warning }
> **Build note:** Most queries below read the primary custom injection table `SocLabEvents_CL` (which carries the S1 identity, S2 email/poisoning, S4 infrastructure, and S5 endpoint commodity signals **plus the ~25 decoy noise event-sets**) alongside Sentinel-backed connectors whose data is injected/replayed for the lab; the S4 infrastructure reach is corroborated by `AzureActivity`. The AI-layer (S3) evidence is the **real Defender for Cloud — AI threat protection alert**, queried via `SecurityAlert` / `AlertEvidence` — not a custom prompt-log table. Exact column names, the ingestion lag, and the **randomized/rebased `TimeGenerated`** values will be finalized during the build (JAM-208). Because the seeder randomizes timestamps and interleaves the chain among the decoys, **widen your time range** generously; if a table returns no rows, confirm the connector/custom table is provisioned before assuming the query is wrong — table availability is environment-dependent, not deterministic.

### 02: Stage 1 — Initial access (impossible-travel sign-in)

Confirm the AiTM-driven sign-in for Mariya from the Lagos IP.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Run the sign-in query for the compromised account. Mariya Petrova (`mariya.petrova@zava.com`) and the attacker IP `102.89.42.17` _(illustrative — confirm at build)_ are the anchors:

   ```kusto
   SigninLogs
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where UserPrincipalName == "mariya.petrova@zava.com"
   | extend City = tostring(LocationDetails.city), Country = tostring(LocationDetails.countryOrRegion)
   | project TimeGenerated, UserPrincipalName, IPAddress, City, Country,
             AppDisplayName, ResultType, RiskLevelDuringSignIn, RiskState
   | order by TimeGenerated asc
   ```

   You should see Mariya's normal Seattle sign-ins and then a successful sign-in from `102.89.42.17` (Lagos, NG) around **08:31** — geographically impossible given the preceding Seattle activity.

2. Corroborate with the risk-event table to confirm Entra ID Protection scored it:

   ```kusto
   AADUserRiskEvents
   | where UserPrincipalName == "mariya.petrova@zava.com"
   | project TimeGenerated, RiskEventType, RiskLevel, IpAddress, Location, DetectionTimingType
   | order by TimeGenerated asc
   ```

   Expect a `impossibleTravel` (or `unfamiliarFeatures`) risk event at **medium** risk. Note the severity — medium is exactly the level that gets auto-triaged during a busy shift.

3. Record Stage 1: **08:31 · Identity · Mariya Petrova / `102.89.42.17` · `SigninLogs` + `AADUserRiskEvents`.**

   {: .note }
   > 📷 **Screenshot to capture:** the `SigninLogs` results showing the Seattle → Lagos impossible-travel sign-in with its risk level.

</details>

### 03: Stage 2 — Poisoning (phishing delivery into the grounding source)

Find the look-alike phishing email that carried the attack in.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Hunt the email tables for the look-alike sender `billing@zava-refunds[.]net` _(illustrative — confirm at build)_ that delivered to Mariya:

   ```kusto
   EmailEvents
   | where SenderFromAddress endswith "zava-refunds.net"
      or RecipientEmailAddress == "mariya.petrova@zava.com"
   | project TimeGenerated, NetworkMessageId, SenderFromAddress, SenderDisplayName,
             RecipientEmailAddress, Subject, DeliveryAction, ThreatTypes, AuthenticationDetails
   | order by TimeGenerated asc
   ```

   Look for the message that landed (`DeliveryAction == "Delivered"`) shortly before the impossible-travel sign-in — the AiTM lure.

2. Pivot to the embedded URL to confirm the credential-proxy link:

   ```kusto
   EmailEvents
   | where SenderFromAddress endswith "zava-refunds.net"
   | join kind=inner EmailUrlInfo on NetworkMessageId
   | project TimeGenerated, SenderFromAddress, RecipientEmailAddress, Subject, Url, UrlDomain
   ```

   The URL domain should match the look-alike infrastructure rather than a legitimate Zava domain.

   {: .note }
   > **On naming this "Poisoning":** the *delivery* email is the identity lure; the **poisoning** of the grounding source happens when the attacker uses Mariya's stolen session to upload `Vendor-Refund-Policy-Update-Q3.docx` into the **Refund Policy Knowledge Base** (`/sites/RefundKB`). For the lab, that poisoning is represented by the **injected Stage-2 event in `SocLabEvents_CL`** plus the doc planted in the grounding store — so query the injected event for the upload, and tie it to the AI-layer probing through the Defender for Cloud AI alert in Step 04. Flag the grounding-source correlation gap in Task 02.03.

3. Record Stage 2: **~08:47 · Data / AI grounding · `billing@zava-refunds[.]net` → `Vendor-Refund-Policy-Update-Q3.docx` in `/sites/RefundKB` · `EmailEvents` / `EmailUrlInfo` (delivery) + injected `SocLabEvents_CL` event (poisoning).**

   {: .note }
   > 📷 **Screenshot to capture:** the `EmailEvents` row for the look-alike sender showing `Delivered`, joined to its `EmailUrlInfo` URL domain.

</details>

### 04: Stage 3 — Reasoning probe (the Defender for Cloud AI alert)

Surface the **real** Defender for Cloud — AI threat protection alert for the prompt-injection / jailbreak attack on the Refund Agent.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. The AI attack against the **real Azure OpenAI** Refund Agent endpoint (`refund-agent-prod`, `gpt-4o`) was a **scripted detonation** that **Microsoft Defender for Cloud — AI threat protection** caught and raised as a **native alert**. Unlike the commodity stages, this evidence is **not** in `SocLabEvents_CL` — it lands in the security-alert tables. Find it via `SecurityAlert`:

   ```kusto
   SecurityAlert
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where ProductName has "Defender for Cloud"
   | where AlertName has_any ("prompt injection", "jailbreak", "AI", "Azure OpenAI")
      or Entities has "refund-agent-prod"
   | project TimeGenerated, AlertName, AlertSeverity, ProductName,
             CompromisedEntity, Entities, ExtendedProperties
   | order by TimeGenerated asc
   ```

   You should see one or more Defender for Cloud AI alerts around the **09:05–09:25** window referencing the `refund-agent-prod` resource — the prompt-injection / jailbreak detection. **This alert fired natively** — the AI layer's problem is not that it was missed, but that it was **never correlated** into the cross-layer incident and sat **buried among the ~25 decoys**.

2. Pull the alert's evidence to confirm intent and tie it to the poisoned grounding and the compromised identity:

   ```kusto
   AlertEvidence
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where ServiceSource has "Defender for Cloud"
      or AdditionalFields has "refund-agent-prod"
   | project TimeGenerated, AlertId, EntityType, EvidenceRole,
             RemoteUrl, AccountUpn, CloudResource, AdditionalFields
   | order by TimeGenerated asc
   ```

   Look in the evidence for the link back to `Vendor-Refund-Policy-Update-Q3.docx` / the Refund Policy Knowledge Base and to Mariya's compromised context — that link is what connects the poisoning stage to the agent's behavior.

   {: .warning }
   > **Build note:** the exact `AlertName`, severity, and `ExtendedProperties`/`AdditionalFields` schema emitted by Defender for Cloud — AI threat protection will be confirmed during the build (JAM-208). The detonation uses the **`openai` package directly** (PyRIT optional — dropped in validation for Python 3.13 / Windows path issues). Treat the *presence of a native Defender for Cloud AI alert on `refund-agent-prod`* as the thing to confirm, not specific alert strings. _(Optional extra: if an M365-Copilot angle is also wired in, a labeled `XPIADetected`/`JailbreakDetected` signal may also appear in `CloudAppEvents` — optional only, never the primary AI source.)_

3. Record Stage 3: **09:05–09:25 · AI · Zava Refund Agent (`refund-agent-prod`) · Defender for Cloud — AI threat protection alert (`SecurityAlert` / `AlertEvidence`).** Flag this as the stage whose alert **fired but was left uncorrelated and buried in the noise** — the gap you'll formalize in Task 02.03.

   {: .note }
   > 📷 **Screenshot to capture:** the `SecurityAlert` row for the Defender for Cloud — AI threat protection alert on `refund-agent-prod`, with its severity and entities.

</details>

### 05: Stage 4 — Lateral movement (anomalous infrastructure requests)

Trace the service principal's move toward the AI/GPU inference infrastructure.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Hunt the injected `SocLabEvents_CL` table for the Stage-4 infrastructure-reach event by `sp-refund-agent-inference` _(illustrative — confirm at build)_ toward the production AI resource group `rg-zava-ai-prod`:

   ```kusto
   SocLabEvents_CL
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where ServicePrincipalName_s == "sp-refund-agent-inference"
      or ResourceGroup_s =~ "rg-zava-ai-prod"
   | project TimeGenerated, ServicePrincipalName_s, OperationName_s,
             ResourceGroup_s, TokenRequests_d, ComputeUnits_d, SourceIp_s
   | order by TimeGenerated asc
   ```

   Look around **09:38** for the anomalous token/compute draw toward the inference infrastructure that doesn't match the service principal's normal pattern.

2. Corroborate the control-plane reach with `AzureActivity`:

   ```kusto
   AzureActivity
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where Caller has "sp-refund-agent-inference"
      or ResourceGroup =~ "rg-zava-ai-prod"
   | project TimeGenerated, Caller, CallerIpAddress, OperationNameValue,
             ResourceGroup, ActivityStatusValue, Properties
   | order by TimeGenerated asc
   ```

   A spike in token requests / compute operations toward `refund-agent-prod` in `rg-zava-ai-prod` (West US 3), corroborated in `AzureActivity`, is the infrastructure-reach signal.

   {: .warning }
   > **Build note (S4 form):** the default build models S4 as the **injected** service-principal infra/inference-reach above. An optional build alternative recasts S4 as **OAuth illicit-consent + inbox rules** (more realistic) — if the build chooses that, hunt the consent grant and mailbox-rule events instead and corroborate via `AzureActivity` / `CloudAppEvents`. Pages stay on the injected default unless the build chooses otherwise.

3. Record Stage 4: **09:38 · Infrastructure · `sp-refund-agent-inference` → `rg-zava-ai-prod` · `SocLabEvents_CL` + `AzureActivity`.** This is the stage you'll confirm as **lateral movement** in Defender for Identity in Task 02.02.

   {: .note }
   > 📷 **Screenshot to capture:** the `SocLabEvents_CL` / `AzureActivity` results showing `sp-refund-agent-inference` operating against `rg-zava-ai-prod` at ~09:38.

</details>

### 06: Stage 5 — Execution (malicious script on the endpoint)

Confirm the encoded PowerShell beacon on the synthetic endpoint OPS-LT-0427.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

{: .note }
> **OPS-LT-0427 is a synthetic asset.** Stage 5 is **injected Sentinel telemetry** — there is no real device and no facilitator-detonated endpoint. The `update_check.ps1` beacon appears only as injected `SocLabEvents_CL` events that mirror what `DeviceProcessEvents` / `DeviceNetworkEvents` would carry.

1. Hunt the injected `SocLabEvents_CL` endpoint events for `update_check.ps1` on OPS-LT-0427 _(illustrative — confirm at build)_:

   ```kusto
   SocLabEvents_CL
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where DeviceName_s == "OPS-LT-0427"
   | where FileName_s in~ ("powershell.exe", "pwsh.exe")
      or ProcessCommandLine_s has "update_check.ps1"
   | project TimeGenerated, DeviceName_s, AccountName_s, FileName_s,
             ProcessCommandLine_s, RemoteIp_s, RemoteUrl_s
   | order by TimeGenerated asc
   ```

   Look around **09:46** for a PowerShell invocation referencing `update_check.ps1`, ideally with an encoded command (`-enc` / `-EncodedCommand`) — the beacon — and the outbound `RemoteIp_s` / `RemoteUrl_s` it contacted.

2. Note the remote IP/URL the beacon contacted — it bounds the endpoint stage's blast radius and gives you an indicator to pivot on across the rest of the injected telemetry.

3. Record Stage 5: **09:46 · Endpoint · OPS-LT-0427 / `update_check.ps1` · injected `SocLabEvents_CL` (synthetic asset).**

   {: .note }
   > 📷 **Screenshot to capture:** the `SocLabEvents_CL` row showing the encoded `update_check.ps1` execution on the synthetic OPS-LT-0427.

</details>
