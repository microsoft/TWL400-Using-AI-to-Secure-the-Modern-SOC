# Task 02.01 — Hunt the unified Sentinel telemetry (KQL) and place each stage

## Introduction

The Exercise 01 narrative told you *what* happened; now you prove it from the data. Microsoft Sentinel is Acesa's unified data backbone — it ingests the identity, email, endpoint, and Azure activity that Defender XDR raised alerts on, plus the custom AI-layer logs that have **no** native detection of their own. Hunting these tables directly does two things alert-driven triage can't: it puts every stage on one timeline at the same fidelity, and it surfaces the AI-layer activity that never became an alert. By the end of this task each of the five attack stages will be placed on the lifecycle and linked back to the entities you anchored in Exercise 01.

## Description

You will run a series of KQL hunting queries in Microsoft Sentinel — one per attack stage — against the Sentinel-backed and custom-seeded tables. For each stage you will confirm the event, pull the timestamp and the entity, and record where it sits on the lifecycle. You will pivot on the two lead entities from Exercise 01 — **Mariya Petrova** (identity) and **`sp-refund-agent-inference`** (infrastructure) — and stitch the results into a single, evidence-backed sequence.

## Success Criteria

- You have run a hunting query per stage and confirmed each event in telemetry, capturing its timestamp and primary entity
- **All five stages placed and linked end to end** _(Validation: all five stages placed and linked end to end)_
- Each placed stage references an entity carried forward from Exercise 01 (Mariya Petrova, `billing@acesa-refunds[.]net`, `102.89.42.17`, the Refund Policy Knowledge Base / `Vendor-Refund-Policy-Update-Q3.docx`, the Acesa Refund Agent, `sp-refund-agent-inference`, OPS-LT-0427)
- You can name which stage produced the **weakest** native telemetry (the AI layer) — the gap you'll formalize in Task 02.03

### 02: Stage 1 — Initial access (impossible-travel sign-in)

Confirm the AiTM-driven sign-in for Mariya from the Lagos IP.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Run the sign-in query for the compromised account. Mariya Petrova (`mariya.petrova@acesa.com`) and the attacker IP `102.89.42.17` _(illustrative — confirm at build)_ are the anchors:

   ```kusto
   SigninLogs
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where UserPrincipalName == "mariya.petrova@acesa.com"
   | extend City = tostring(LocationDetails.city), Country = tostring(LocationDetails.countryOrRegion)
   | project TimeGenerated, UserPrincipalName, IPAddress, City, Country,
             AppDisplayName, ResultType, RiskLevelDuringSignIn, RiskState
   | order by TimeGenerated asc
   ```

   You should see Mariya's normal Seattle sign-ins and then a successful sign-in from `102.89.42.17` (Lagos, NG) around **08:31** — geographically impossible given the preceding Seattle activity.

2. Corroborate with the risk-event table to confirm Entra ID Protection scored it:

   ```kusto
   AADUserRiskEvents
   | where UserPrincipalName == "mariya.petrova@acesa.com"
   | project TimeGenerated, RiskEventType, RiskLevel, IpAddress, Location, DetectionTimingType
   | order by TimeGenerated asc
   ```

   Expect a `impossibleTravel` (or `unfamiliarFeatures`) risk event at **medium** risk. Note the severity — medium is exactly the level that gets auto-triaged during a busy shift.

3. Record Stage 1: **08:31 · Identity · Mariya Petrova / `102.89.42.17` · `SigninLogs` + `AADUserRiskEvents`.**

   {: .note }
   > 📷 **Screenshot to capture:** the `SigninLogs` results showing the Seattle → Lagos impossible-travel sign-in with its risk level.

</details>

### 03: Stage 2 — Poisoning (phishing delivery into the grounding source)

Find the look-alike phishing email that carried the attack in.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Hunt the email tables for the look-alike sender `billing@acesa-refunds[.]net` _(illustrative — confirm at build)_ that delivered to Mariya:

   ```kusto
   EmailEvents
   | where SenderFromAddress endswith "acesa-refunds.net"
      or RecipientEmailAddress == "mariya.petrova@acesa.com"
   | project TimeGenerated, NetworkMessageId, SenderFromAddress, SenderDisplayName,
             RecipientEmailAddress, Subject, DeliveryAction, ThreatTypes, AuthenticationDetails
   | order by TimeGenerated asc
   ```

   Look for the message that landed (`DeliveryAction == "Delivered"`) shortly before the impossible-travel sign-in — the AiTM lure.

2. Pivot to the embedded URL to confirm the credential-proxy link:

   ```kusto
   EmailEvents
   | where SenderFromAddress endswith "acesa-refunds.net"
   | join kind=inner EmailUrlInfo on NetworkMessageId
   | project TimeGenerated, SenderFromAddress, RecipientEmailAddress, Subject, Url, UrlDomain
   ```

   The URL domain should match the look-alike infrastructure rather than a legitimate Acesa domain.

   {: .note }
   > **On naming this "Poisoning":** the *delivery* email is the identity lure; the **poisoning** of the grounding source happens when the attacker uses Mariya's stolen session to upload `Vendor-Refund-Policy-Update-Q3.docx` into the **Refund Policy Knowledge Base** (`/sites/RefundKB`). That upload is **SharePoint/Purview-side** and is not reliably represented in the Sentinel-backed tables — capture it from the Exercise 01 narrative and Purview DSPM for AI, and flag the telemetry gap in Task 02.03.

3. Record Stage 2: **~08:47 · Data / AI grounding · `billing@acesa-refunds[.]net` → `Vendor-Refund-Policy-Update-Q3.docx` in `/sites/RefundKB` · `EmailEvents` / `EmailUrlInfo` (delivery) + DSPM for AI (poisoned content).**

   {: .note }
   > 📷 **Screenshot to capture:** the `EmailEvents` row for the look-alike sender showing `Delivered`, joined to its `EmailUrlInfo` URL domain.

</details>

### 04: Stage 3 — Reasoning probe (prompt-injection volume spike)

Detect the burst of prompt-injection / jailbreak attempts against the Refund Agent.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. The Refund Agent's prompt traffic lands in the custom-seeded table `RefundAgentPromptLogs_CL` _(illustrative — confirm at build)_. Summarize request volume into 5-minute bins to expose the spike:

   ```kusto
   RefundAgentPromptLogs_CL
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | summarize Requests = count(),
               Blocked = countif(PromptShieldVerdict_s == "blocked"),
               DistinctSessions = dcount(SessionId_g)
       by bin(TimeGenerated, 5m)
   | order by TimeGenerated asc
   ```

   You should see baseline traffic step up sharply between **09:05 and 09:25** — a volume and pattern unlike normal staff usage.

2. Inspect the offending prompts to confirm intent (jailbreak / injection wording) and tie them to the poisoned grounding:

   ```kusto
   RefundAgentPromptLogs_CL
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where PromptShieldVerdict_s in ("blocked", "flagged")
      or PromptText_s has_any ("ignore previous", "system prompt", "developer mode", "exfiltrate")
   | project TimeGenerated, SessionId_g, UserPrincipalName_s, PromptText_s,
             PromptShieldVerdict_s, GroundingDocument_s
   | order by TimeGenerated asc
   ```

   Note whether `GroundingDocument_s` references `Vendor-Refund-Policy-Update-Q3.docx` — that link is what connects the poisoning stage to the agent's behavior.

   {: .warning }
   > **Build note:** `RefundAgentPromptLogs_CL` is custom-seeded for the lab and (per the recommended build) the AI-layer signals may be **detonated live with PyRIT** rather than replayed. Volume, exact verdict strings, and prompt text will vary run to run. Treat the *shape* (a clear spike of injection attempts in the 09:05–09:25 window) as the thing to confirm, not specific counts.

3. Record Stage 3: **09:05–09:25 · AI · Acesa Refund Agent (`refund-agent-prod`) · `RefundAgentPromptLogs_CL`.** Flag this as the stage with the **weakest native detection** — there's no out-of-the-box alert behind it.

   {: .note }
   > 📷 **Screenshot to capture:** the 5-minute-bin summary chart showing the prompt-volume spike across the 09:05–09:25 window.

</details>

### 05: Stage 4 — Lateral movement (anomalous infrastructure requests)

Trace the service principal's move toward the AI/GPU inference infrastructure.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Hunt `AzureActivity` for actions performed by `sp-refund-agent-inference` _(illustrative — confirm at build)_ against the production AI resource group `rg-acesa-ai-prod`:

   ```kusto
   AzureActivity
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where Caller has "sp-refund-agent-inference"
      or ResourceGroup =~ "rg-acesa-ai-prod"
   | project TimeGenerated, Caller, CallerIpAddress, OperationNameValue,
             ResourceGroup, ActivityStatusValue, Properties
   | order by TimeGenerated asc
   ```

   Look around **09:38** for token requests and compute/deployment operations against the inference infrastructure that don't match the service principal's normal pattern.

2. Corroborate with the custom inference-audit table for anomalous token/compute draw:

   ```kusto
   AIInferenceAudit_CL
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where ServicePrincipalName_s == "sp-refund-agent-inference"
   | summarize TokenRequests = count(), TotalComputeUnits = sum(ComputeUnits_d)
       by bin(TimeGenerated, 5m), OperationName_s
   | order by TimeGenerated asc
   ```

   A spike in token requests / compute units toward `refund-agent-prod` in `rg-acesa-ai-prod` (West US 3) is the infrastructure-reach signal.

3. Record Stage 4: **09:38 · Infrastructure · `sp-refund-agent-inference` → `rg-acesa-ai-prod` · `AzureActivity` + `AIInferenceAudit_CL`.** This is the stage you'll confirm as **lateral movement** in Defender for Identity in Task 02.02.

   {: .note }
   > 📷 **Screenshot to capture:** the `AzureActivity` results showing `sp-refund-agent-inference` operating against `rg-acesa-ai-prod` at ~09:38.

</details>

### 06: Stage 5 — Execution (malicious script on the endpoint)

Confirm the encoded PowerShell beacon running on Mariya's laptop.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Hunt `DeviceProcessEvents` for `update_check.ps1` on OPS-LT-0427 _(illustrative — confirm at build)_:

   ```kusto
   DeviceProcessEvents
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where DeviceName == "OPS-LT-0427"
   | where FileName in~ ("powershell.exe", "pwsh.exe")
      or ProcessCommandLine has "update_check.ps1"
   | project TimeGenerated, DeviceName, AccountName, FileName,
             ProcessCommandLine, InitiatingProcessFileName, InitiatingProcessCommandLine
   | order by TimeGenerated asc
   ```

   Look around **09:46** for a PowerShell invocation referencing `update_check.ps1`, ideally with an encoded command (`-enc` / `-EncodedCommand`) — the beacon.

2. Pivot to the outbound beacon to bound what the execution reached out to:

   ```kusto
   DeviceNetworkEvents
   | where TimeGenerated between (datetime(2026-01-01) .. datetime(2026-12-31))
   | where DeviceName == "OPS-LT-0427"
   | where InitiatingProcessFileName in~ ("powershell.exe", "pwsh.exe")
   | project TimeGenerated, DeviceName, RemoteIP, RemoteUrl, RemotePort,
             InitiatingProcessFileName, InitiatingProcessCommandLine
   | order by TimeGenerated asc
   ```

   Note the remote IP/URL the beacon contacted — it bounds the endpoint stage's blast radius and gives you an indicator to sweep other devices for.

3. Record Stage 5: **09:46 · Endpoint · OPS-LT-0427 / `update_check.ps1` · `DeviceProcessEvents` + `DeviceNetworkEvents`.**

   {: .note }
   > 📷 **Screenshot to capture:** the `DeviceProcessEvents` row showing the encoded `update_check.ps1` execution on OPS-LT-0427.

</details>

Stitch the confirmed events into a single end-to-end sequence linked to entities.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

   | # | Stage | Time | Layer | Entity _(illustrative — confirm at build)_ | Evidenced by |

   | 1 | Initial access | 08:31 | Identity | Mariya Petrova / `102.89.42.17` | `SigninLogs`, `AADUserRiskEvents` |
