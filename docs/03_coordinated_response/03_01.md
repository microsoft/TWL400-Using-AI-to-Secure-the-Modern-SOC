# Task 03.01 — Generate the response with Copilot and execute coordinated containment

## Introduction

You have a complete, evidence-backed lifecycle from Exercise 02. Now you contain it. The instinct in a traditional SOC is to fix the loudest layer first — usually the endpoint — and circle back to the rest. But this attack is a single chain across three layers, and containing one while leaving the others open just buys the attacker time on the layers you ignored. Revoking Mariya's session does nothing if the poisoned document is still grounding the Refund Agent; quarantining the document does nothing if the service principal can still reach the inference infrastructure.

So you start by asking **Security Copilot** to recommend a coordinated response across every layer at once, then you execute it layer by layer — identity, data/AI, and infrastructure — knowing each action is one part of a single containment.

## Description

You will prompt Security Copilot to generate a recommended, cross-layer response to the reconstructed attack, then execute coordinated containment across all three affected layers: revoke Mariya Petrova's sessions and disable the service principal (identity), quarantine the poisoned document and scope the AI exposure with Purview DSPM for AI (data/AI), and restrict the affected inference deployment via a generated playbook (infrastructure).

## Success Criteria

- Security Copilot has produced a recommended coordinated response that addresses identity, data/AI, and infrastructure _(Validation: Copilot-generated response reviewed)_
- **Containment spans identity, data, and infrastructure layers** _(Validation: containment spans identity, data, and infrastructure layers)_
- Mariya Petrova's sessions are revoked / reauthentication is forced, and the service principal `sp-refund-agent-inference` is disabled and its credentials rotated
- The poisoned document `Vendor-Refund-Policy-Update-Q3.docx` is quarantined, and the impacted Refund Agent interactions are identified in Purview DSPM for AI
- The affected inference deployment (`refund-agent-prod` in `rg-acesa-ai-prod`) is isolated/restricted via a generated playbook

## Key Tasks

### 01: Ask Security Copilot for a coordinated response

Get the machine-speed recommendation before you touch any one console.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Open standalone Security Copilot at [https://securitycopilot.microsoft.com](https://securitycopilot.microsoft.com){:target="_blank"}, or continue the session you used in Exercise 01 so it keeps the incident context.

2. Submit a prompt that asks for a coordinated, cross-layer response — not a single action. Replace `<INCIDENT_ID>` with the incident ID from Task 01.01:

   ```text
   For Microsoft Defender incident <INCIDENT_ID>, recommend a coordinated
   containment response for the reconstructed cross-layer attack on the Acesa
   Refund Agent. Organize the response by layer — identity, data/AI, and
   infrastructure — and for each recommended action name the target entity, the
   Microsoft product to perform it in, and whether the action is reversible.
   Cover, at minimum: the compromised user Mariya Petrova, the service principal
   sp-refund-agent-inference, the poisoned document
   Vendor-Refund-Policy-Update-Q3.docx, and the inference deployment
   refund-agent-prod in rg-acesa-ai-prod.
   ```

3. Review the recommendation critically. A good response groups the actions the way you will execute them below:

   | Layer | Recommended action | Target entity | Product |
   |-------|--------------------|---------------|---------|
   | Identity | Revoke sessions / force reauthentication | Mariya Petrova | Entra Conditional Access |
   | Identity | Disable and rotate credentials | `sp-refund-agent-inference` | Microsoft Entra |
   | Data / AI | Quarantine the poisoned grounding document | `Vendor-Refund-Policy-Update-Q3.docx` | Microsoft Purview |
   | Data / AI | Identify impacted Refund Agent interactions | Refund Policy Knowledge Base | Purview DSPM for AI |
   | Infrastructure | Isolate / restrict the inference deployment | `refund-agent-prod` (`rg-acesa-ai-prod`) | Sentinel playbook |

   {: .note }
   > 📷 **Screenshot to capture:** the Security Copilot response listing the recommended coordinated actions grouped by identity, data/AI, and infrastructure layer.

   {: .warning }
   > Security Copilot responses are non-deterministic — the exact wording and ordering will vary between runs. What must be present is a recommendation that spans **all three layers**. If a run fixes only the identity or endpoint layer and drops the AI/data and infrastructure actions, re-prompt explicitly for the missing layers. Supervising the recommendation is itself part of the lesson.

</details>

### 02: Contain the identity layer

Cut the attacker out of the identity that started the chain.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. **Revoke Mariya Petrova's sessions.** In the Microsoft Entra admin center, go to **Identity > Users > All users**, open **Mariya Petrova** (`mariya.petrova@acesa.com`), and select **Revoke sessions**. This invalidates her refresh tokens so the stolen session can no longer be replayed.

2. **Force reauthentication going forward.** Confirm the Conditional Access policy that requires reauthentication / strong (phishing-resistant) MFA applies to Mariya — the policy **`CA-Refund-Ops-Reauth`** _(illustrative — confirm at build)_ in **Protection > Conditional Access**. When she next signs in, she must reauthenticate with MFA from a compliant device, closing the AiTM-stolen-token path identified in Exercise 02.

3. **Disable and rotate the service principal.** In **Identity > Applications > Enterprise applications** (or **App registrations**), open `sp-refund-agent-inference`. Set **Enabled for users to sign in** to **No** to disable it, then under **Certificates & secrets** remove the compromised secret/certificate and add a fresh one for the legitimate workload. This stops the anomalous token/compute requests you traced to this principal in Exercise 02 (Stage 4).

   {: .note }
   > 📷 **Screenshot to capture:** the Entra user blade for Mariya Petrova showing **Revoke sessions**, and the `sp-refund-agent-inference` enterprise application set to disabled.

   {: .warning }
   > Whether disabling the service principal is safe to run autonomously, or whether it needs human sign-off because a legitimate workload depends on it, is an open design decision — you will settle it in Task 03.03 and it must be confirmed against the workload's real dependencies at build.

</details>

### 03: Contain the data / AI layer

Stop the poisoned grounding from shaping any further Refund Agent answers, and find out how far it already reached.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. **Quarantine the poisoned document.** In the Refund Policy Knowledge Base SharePoint site (`/sites/RefundKB`), locate `Vendor-Refund-Policy-Update-Q3.docx` and move it out of the grounding scope — either delete it or move it to a quarantine library that the Refund Agent's indexer does not crawl. Removing it from the grounded content stops it from being retrieved into future Refund Agent answers.

2. **Confirm it leaves the grounding index.** Trigger (or wait for) a re-index of the knowledge base so the quarantined document is dropped from the Refund Agent's retrievable corpus. Until the index refreshes, treat the agent's grounding as still potentially poisoned.

3. **Scope the AI exposure with Purview DSPM for AI.** In the Microsoft Purview portal, open **DSPM for AI** and review the AI interaction activity for the Refund Agent. Use the activity/audit view to identify **which Refund Agent interactions drew on the poisoned grounding** — the prompts and responses that retrieved `Vendor-Refund-Policy-Update-Q3.docx` or the Refund Policy Knowledge Base during the poisoning window (from 08:47 onward, per the Exercise 02 timeline). Record the impacted interactions; you will use this list to bound residual risk in Task 03.04.

   ```text
   Using Purview DSPM for AI, list the Acesa Refund Agent interactions between
   08:47 and now whose grounding/citations referenced
   Vendor-Refund-Policy-Update-Q3.docx or the Refund Policy Knowledge Base.
   For each, give the timestamp, the user, and a one-line summary of the response.
   ```

   {: .note }
   > 📷 **Screenshot to capture:** the Purview DSPM for AI activity view showing Refund Agent interactions that referenced the poisoned grounding source, with timestamps.

   {: .note }
   > The interaction-level grounding view in DSPM for AI is the piece commodity triage never has — it is the only way to answer "what did the poisoned document actually change?" Treat this list as the blast radius for the AI layer.

</details>

### 04: Contain the infrastructure layer

Restrict the inference deployment the attacker moved toward — the most business-impacting action, so generate it as a playbook rather than clicking it live.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. **Generate the containment playbook.** Ask Security Copilot to draft the infrastructure-layer action as a Sentinel playbook (Logic App) rather than performing it ad hoc:

   ```text
   Draft a Microsoft Sentinel playbook (Logic App) named Acesa-Contain-CrossLayer
   that isolates the affected AI inference deployment refund-agent-prod in resource
   group rg-acesa-ai-prod: restrict inbound network access to the inference
   endpoint, revoke the deployment's access to the compromised service principal,
   and post a containment summary to the incident. Include a manual-approval step
   before the deployment is isolated.
   ```

2. **Review and stage — do not blindly run.** The generated playbook isolates/restricts `refund-agent-prod` (for example, locking the endpoint behind a deny rule or disabling the deployment). Because isolating production inference can take the Refund Agent offline for legitimate users, keep the **manual-approval step** in the flow. You will wire this playbook to an automation rule in Task 03.02 and decide its autonomy in Task 03.03.

3. **Apply the restriction with approval.** Trigger the playbook (or, for this lab, apply the equivalent restriction in Defender for Cloud / the Azure portal against `refund-agent-prod`) and approve the isolation step. Confirm the inference endpoint no longer accepts requests from the disabled service principal and that inbound access is restricted.

   {: .note }
   > 📷 **Screenshot to capture:** the generated `Acesa-Contain-CrossLayer` playbook in the Logic Apps designer, highlighting the manual-approval step before the isolate-deployment action.

   {: .warning }
   > Whether this playbook hard-isolates the deployment or only restricts it to known-good callers — and how deep the autonomy goes — is an open design decision pending the tenant build. Confirm the exact isolation mechanism for `refund-agent-prod` against the provisioned environment.

</details>

## Summary

You asked Security Copilot to recommend a single coordinated response instead of a pile of disconnected tickets, then executed it across all three layers the attack touched: you cut the attacker out of the identity (revoked Mariya's sessions, disabled and rotated `sp-refund-agent-inference`), stopped and scoped the AI-layer damage (quarantined the poisoned document and used Purview DSPM for AI to find the impacted interactions), and restricted the inference infrastructure (`refund-agent-prod`) through a generated playbook. Containment now spans identity, data, and infrastructure. Next, you'll operationalize this response in Microsoft Sentinel so it runs as automation rather than by hand.
