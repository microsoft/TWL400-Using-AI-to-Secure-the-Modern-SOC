# Task 03.01 — Generate the response with Copilot and execute coordinated containment

## Introduction

You have a complete, evidence-backed lifecycle from Exercise 02. Now you contain it. The instinct in a traditional SOC is to fix the loudest layer first — usually the endpoint — and circle back to the rest. But this attack is a single chain across three layers, and containing one while leaving the others open just buys the attacker time on the layers you ignored. Revoking Mariya's session does nothing if the poisoned document is still grounding the Refund Agent; quarantining the document does nothing if the service principal can still reach the inference infrastructure.

So you start by asking **Security Copilot** to recommend a coordinated response across every layer at once, then you execute it layer by layer — identity, data/AI, and infrastructure — knowing each action is one part of a single containment.

## Description

You will prompt Security Copilot to generate a recommended, cross-layer response to the reconstructed attack, then execute coordinated containment across all three affected layers: revoke Mariya Petrova's sessions and disable the service principal (identity), quarantine the poisoned document and scope the AI exposure with Microsoft Defender for Cloud — AI threat protection (data/AI), and restrict the affected inference deployment via a generated playbook (infrastructure).

## Success Criteria

- Security Copilot has produced a recommended coordinated response that addresses identity, data/AI, and infrastructure _(Validation: Copilot-generated response reviewed)_
- **Containment spans identity, data, and infrastructure layers** _(Validation: containment spans identity, data, and infrastructure layers)_
- Mariya Petrova's sessions are revoked / reauthentication is forced, and the service principal `sp-refund-agent-inference` is disabled and its credentials rotated
- The poisoned document `trusted-report.docx` is quarantined from the `grounding` container in storage account `stsoclabtgu35glfngjhc`, and the impacted Refund Agent activity is scoped from the Defender for Cloud — AI threat protection alert (`SecurityAlert` / `ExtendedProperties`) together with the injected Stage-2 event
- The affected Azure OpenAI resource (`oai-soclab-v02` in `rg-soclab`) is isolated/restricted via a generated playbook

## Key Tasks

### 01: Ask Security Copilot for a coordinated response

Get the machine-speed recommendation before you touch any one console.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Open standalone Security Copilot at [https://securitycopilot.microsoft.com](https://securitycopilot.microsoft.com){:target="_blank"}, or continue the session you used in Exercise 01 so it keeps the incident context.

2. Submit a prompt that asks for a coordinated, cross-layer response — not a single action. Replace `<INCIDENT_ID>` with the incident ID from Task 01.01:

   ```text
   For Microsoft Defender incident <INCIDENT_ID>, recommend a coordinated
   containment response for the reconstructed cross-layer attack on the Zava
   Refund Agent. Organize the response by layer — identity, data/AI, and
   infrastructure — and for each recommended action name the target entity, the
   Microsoft product to perform it in, and whether the action is reversible.
   Cover, at minimum: the compromised user Mariya Petrova, the service principal
   sp-refund-agent-inference, the poisoned document trusted-report.docx in the
   grounding container of storage account stsoclabtgu35glfngjhc, and the Azure
   OpenAI resource oai-soclab-v02 in rg-soclab.
   ```

3. Review the recommendation critically. A good response groups the actions the way you will execute them below:

   | Layer | Recommended action | Target entity | Product |
   |-------|--------------------|---------------|---------|
   | Identity | Revoke sessions / force reauthentication | Mariya Petrova | Entra Conditional Access |
   | Identity | Disable and rotate credentials | `sp-refund-agent-inference` | Microsoft Entra |
   | Data / AI | Quarantine the poisoned grounding document | `trusted-report.docx` (blob in `grounding` container, `stsoclabtgu35glfngjhc`) | Azure Storage |
   | Data / AI | Scope impacted Refund Agent activity | Zava Refund Agent / `oai-soclab-v02` | Defender for Cloud — AI threat protection |
   | Infrastructure | Isolate / restrict the inference deployment | `oai-soclab-v02` (`rg-soclab`) | Sentinel playbook |

   {: .note }
   > 📷 **Screenshot to capture:** the Security Copilot response listing the recommended coordinated actions grouped by identity, data/AI, and infrastructure layer.

   {: .warning }
   > Security Copilot responses are non-deterministic — the exact wording and ordering will vary between runs. What must be present is a recommendation that spans **all three layers**. If a run fixes only the identity or endpoint layer and drops the AI/data and infrastructure actions, re-prompt explicitly for the missing layers. Supervising the recommendation is itself part of the lesson.

</details>

### 02: Contain the identity layer

Cut the attacker out of the identity that started the chain.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. **Revoke Mariya Petrova's sessions.** In the Microsoft Entra admin center, go to **Identity > Users > All users**, open **Mariya Petrova** (`mariya.petrova@@lab.Variable(TenantDomain)`), and select **Revoke sessions**. This invalidates her refresh tokens so the stolen session can no longer be replayed.

2. **Force reauthentication going forward.** Confirm the Conditional Access policy that requires reauthentication / strong (phishing-resistant) MFA applies to Mariya — the policy **`CA-Refund-Ops-Reauth`** _(illustrative — confirm at build)_ in **Protection > Conditional Access**. When she next signs in, she must reauthenticate with MFA from a compliant device, closing the AiTM-stolen-token path identified in Exercise 02.

3. **Disable and rotate the service principal.** In **Identity > Applications > Enterprise applications** (or **App registrations**), open `sp-refund-agent-inference`. Set **Enabled for users to sign in** to **No** to disable it, then under **Certificates & secrets** remove the compromised secret/certificate and add a fresh one for the legitimate workload. This stops the anomalous token/compute requests you traced to this principal in Exercise 02 (Stage 4).

   {: .note }
   > 📷 **Screenshot to capture:** the Entra user blade for Mariya Petrova showing **Revoke sessions**, and the `sp-refund-agent-inference` enterprise application set to disabled.

   {: .warning }
   > Whether disabling the service principal is safe to run autonomously, or whether it needs human sign-off because a legitimate workload depends on it, is an open design decision — you will settle it in Task 03.03 and it must be confirmed against the workload's real dependencies at build.

</details>

### 03: Contain the data / AI layer

Stop the poisoned grounding from shaping any further Refund Agent answers, and find out how far it already reached.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. **Quarantine the poisoned document.** The Refund Agent's grounding source is the `grounding` container in Azure Blob Storage (storage account `stsoclabtgu35glfngjhc`, resource group `rg-soclab`). In the Azure portal, navigate to `rg-soclab → stsoclabtgu35glfngjhc → Containers → grounding`, locate `trusted-report.docx`, and delete it (or move it to a separate quarantine container the agent does not read from). Removing it from the grounding container stops it from being retrieved into future Refund Agent answers.

2. **Confirm the blob is gone.** Refresh the container view and confirm `trusted-report.docx` no longer appears. Because the Refund Agent reads blobs at query time rather than from a cached index, removal takes effect immediately for subsequent queries — there is no separate index flush step required.

3. **Scope the AI exposure with Defender for Cloud — AI threat protection.** Open the **Microsoft Defender for Cloud — AI threat protection** alert for the Zava Refund Agent (the real native alert raised on `oai-soclab-v02`, surfaced in the unified incident and queryable via `SecurityAlert`). The alert title reads **"A Jailbreak attempt on your Azure AI model deployment was blocked by Prompt Shields"** — note the word *blocked*. This is accurate: Prompt Shields intercepted the attacker's direct prompt-injection queries at the API layer. However, *blocked* applies **only to the direct injection attempts**; the grounding poisoning (Stage 2) is a separate, upstream vector that Prompt Shields does not evaluate at blob-upload time. Refund Agent answers produced from the poisoned grounding during the window between the Stage 2 upload and containment may still have been compromised — that is the blast radius you are bounding here.

   Read the alert's `ExtendedProperties` to see the attacked endpoint, the number of blocked attempts, and the time window — note that the `AlertEvidence` table is not populated for this alert type in the lab environment; all evidence is in `ExtendedProperties`. Correlate the alert's time window against the **injected Stage-2 poisoning event** (the delivery of `trusted-report.docx` into the `grounding` container, captured in `SocLabEvents_CL` as the Stage 2 `PhishingEmailDelivered` action in your Exercise 02 timeline). Together they bound **which Refund Agent activity overlapped the poisoned grounding** — even though direct injections were blocked, answers produced from poisoned grounding during that window are still in scope. Record the impacted window; you will use it to bound residual risk in Task 03.04.

   ```text
   For Microsoft Defender incident <INCIDENT_ID>, summarize the Defender for Cloud
   AI threat protection alert on the Zava Refund Agent (oai-soclab-v02):
   the attacked Azure OpenAI endpoint, the prompt-injection/jailbreak evidence,
   and the time window. Then correlate that window with the injected Stage-2
   poisoning event for trusted-report.docx (the grounding storage upload captured
   in SocLabEvents_CL) to scope which Refund Agent activity overlapped the
   poisoned grounding.
   ```

   {: .note }
   > 📷 **Screenshot to capture:** the Defender for Cloud — AI threat protection alert and its `ExtendedProperties` for `oai-soclab-v02`, alongside the Stage-2 poisoning event from `SocLabEvents_CL`.

   {: .note }
   > The Defender for Cloud AI alert plus the injected Stage-2 event is what commodity triage never correlates — together they answer "the AI was attacked **and** its grounding was poisoned in the same window." *Blocked* does not mean *safe*: the direct injections were stopped by Prompt Shields, but grounding-poisoned answers produced before containment are still in scope. That overlap is the blast radius for the AI layer; the alert fired, but nothing in the noisy queue had tied it to the poisoning until you did.

</details>

### 04: Contain the infrastructure layer

Restrict the inference deployment the attacker moved toward — the most business-impacting action, so generate it as a playbook rather than clicking it live.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. **Generate the containment playbook.** Ask Security Copilot to draft the infrastructure-layer action as a Sentinel playbook (Logic App) rather than performing it ad hoc:

   ```text
   Draft a Microsoft Sentinel playbook (Logic App) named Zava-Contain-CrossLayer
   that isolates the affected Azure OpenAI resource oai-soclab-v02 in resource
   group rg-soclab: restrict inbound network access to the inference endpoint,
   revoke the deployment's access to the compromised service principal, and post
   a containment summary to the incident. Include a manual-approval step before
   the deployment is isolated.
   ```

2. **Review and stage — do not blindly run.** The generated playbook isolates/restricts `oai-soclab-v02` (for example, locking the endpoint behind a deny rule or disabling the deployment). Because isolating production inference can take the Refund Agent offline for legitimate users, keep the **manual-approval step** in the flow. You will wire this playbook to an automation rule in Task 03.02 and decide its autonomy in Task 03.03.

3. **Apply the restriction with approval.** Trigger the playbook (or, for this lab, apply the equivalent restriction in Defender for Cloud / the Azure portal against `oai-soclab-v02` in `rg-soclab`) and approve the isolation step. Confirm the inference endpoint no longer accepts requests from the disabled service principal and that inbound access is restricted.

   {: .note }
   > 📷 **Screenshot to capture:** the generated `Zava-Contain-CrossLayer` playbook in the Logic Apps designer, highlighting the manual-approval step before the isolate-deployment action.

   {: .warning }
   > Whether this playbook hard-isolates the deployment or only restricts it to known-good callers — and how deep the autonomy goes — is an open design decision pending the tenant build. Confirm the exact isolation mechanism for `oai-soclab-v02` against the provisioned environment.

   {: .note }
   > **Optional build alternative (S4-OAuth):** if the build adopts the OAuth illicit-consent + inbox-rule lateral-movement option for Stage 4 instead of the injected service-principal infra reach, the infrastructure-containment action changes accordingly — revoke the malicious OAuth grant and remove the attacker's inbox rules rather than isolating `refund-agent-prod`. The injected-infra default above stands unless the build chooses the OAuth recast.

</details>

## Summary

You asked Security Copilot to recommend a single coordinated response instead of a pile of disconnected tickets, then executed it across all three layers the attack touched: you cut the attacker out of the identity (revoked Mariya's sessions, disabled and rotated `sp-refund-agent-inference`), stopped and scoped the AI-layer damage (quarantined the poisoned document and used the Defender for Cloud — AI threat protection alert with the injected Stage-2 event to scope the impacted Refund Agent activity), and restricted the inference infrastructure (`oai-soclab-v02` in `rg-soclab`) through a generated playbook. Containment now spans identity, data, and infrastructure. Next, you'll operationalize this response in Microsoft Sentinel so it runs as automation rather than by hand.
