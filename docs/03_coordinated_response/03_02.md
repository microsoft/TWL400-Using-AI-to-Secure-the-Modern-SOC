# Task 03.02 — Operationalize the response with Sentinel automation

## Introduction

In Task 03.01 you executed the coordinated response by hand, layer by layer. That works once, with you driving — but the whole point of an AI-driven SOC is that this same response fires **at machine speed** the next time the chain appears, without waiting for an analyst to be awake and looking. Roughly 75 minutes of attacker dwell time in this scenario is, in large part, the cost of manual, console-by-console response.

In this task you operationalize the response: you wire the coordinated containment into **Microsoft Sentinel automation** so the `Zava-Contain-CrossLayer` playbook runs against this incident through an automation rule, with the human-approval gate preserved where it matters.

## Description

You will configure (or trigger) a Microsoft Sentinel automation rule that runs the `Zava-Contain-CrossLayer` playbook (Logic App) when this multi-stage incident is created or updated, and confirm the playbook performs the coordinated cross-layer containment.

## Success Criteria

- **Sentinel automation is configured/triggered for the response** _(Validation: Sentinel automation configured/triggered for the response)_
- An automation rule binds the multi-stage incident to the `Zava-Contain-CrossLayer` playbook
- A run of the playbook is observed completing its containment actions (with the manual-approval step honored for the business-impacting action)
- The incident records the automation/playbook activity in its activity log

## Key Tasks

### 01: Review the containment playbook

Confirm what you are about to operationalize.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. In the Microsoft Sentinel (or unified SecOps) portal, go to **Configuration > Automation** and open the **Active playbooks** tab. Locate the Logic App **`Zava-Contain-CrossLayer`** _(illustrative — confirm at build)_ — the playbook generated in Task 03.01.

2. Open it in the Logic Apps designer and confirm it performs the coordinated containment across layers, for example:

   | Step | Layer | Action |
   |------|-------|--------|
   | 1 | Identity | Revoke the user's sessions / require reauthentication (Mariya Petrova) |
   | 2 | Identity | Disable the service principal `sp-refund-agent-inference` |
   | 3 | Data / AI | Quarantine `trusted-report.docx` from the `grounding` container |
   | 4 | — | **Manual approval** gate |
   | 5 | Infrastructure | Isolate/restrict `oai-soclab-v02` in `rg-soclab` |
   | 6 | — | Post a containment summary comment back to the incident |

3. Confirm the **manual-approval step** sits before the infrastructure isolation, so the autonomous portion (steps 1–3) runs immediately while the business-impacting step (step 5) waits for a human.

   {: .note }
   > 📷 **Screenshot to capture:** the `Zava-Contain-CrossLayer` Logic App in the designer, showing the layered containment steps and the manual-approval gate before the isolate-deployment action.

   {: .warning }
   > The split between the auto-run steps and the approval-gated step encodes the autonomy boundary you define in Task 03.03. If the build resolves that boundary differently, update the playbook's approval placement to match — the canon for that decision lives in Task 03.03.

</details>

### 02: Create the automation rule

Bind the playbook to the incident so it runs without a human kicking it off.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. In **Configuration > Automation**, select **Create > Automation rule**.

2. Configure the rule **`AR-Contain-CrossLayer`** _(illustrative — confirm at build)_:

   - **Trigger:** *When incident is created* (and, optionally, *When incident is updated*).
   - **Conditions:** scope to this attack — for example **Title** contains *"Multi-stage incident involving Identity"* (the stable substring of the full title *"SOC Lab — cross-layer attack"*), or **Tactics** include lateral movement / credential access. Keep the condition tight so the playbook only fires on this cross-layer pattern, not on every incident.
   - **Actions:** *Run playbook* → select **`Zava-Contain-CrossLayer`**.

3. Confirm Sentinel has permission to run the playbook (the Logic App's managed identity / the Sentinel automation account must be granted the **Microsoft Sentinel Automation Contributor** role on the playbook's resource group). Save the rule.

   {: .note }
   > 📷 **Screenshot to capture:** the automation rule editor showing the trigger, the incident-title condition, and the **Run playbook** action pointing at `Zava-Contain-CrossLayer`.

</details>

### 03: Trigger and verify

Watch the response run as automation.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Trigger the automation. Either re-run the rule against the existing incident (from the incident's **Actions > Run playbook** menu) or let the **When incident is created/updated** trigger fire. When the playbook reaches the **manual-approval** step, approve the infrastructure isolation.

2. Verify the run. In the playbook's **Run history** (Logic Apps) confirm the run succeeded, and on the incident's **Activity log** confirm the playbook posted its containment summary. Spot-check that the autonomous steps actually took effect — Mariya's sessions revoked, the service principal disabled, the document quarantined — consistent with what you did manually in Task 03.01.

   {: .note }
   > 📷 **Screenshot to capture:** the playbook run history showing a successful run, and the incident activity log entry posted by `Zava-Contain-CrossLayer`.

   {: .tip }
   > This is the difference Module 4 builds on: the same coordinated containment you ran by hand in Task 03.01 now executes from a single incident trigger, in seconds, with one human approval instead of six manual console visits.

</details>

## Summary

You operationalized the coordinated response: the `Zava-Contain-CrossLayer` playbook now runs from a Sentinel automation rule bound to the multi-stage incident, executing the identity and data/AI containment autonomously and gating the business-impacting infrastructure isolation behind a human approval. The response is no longer a manual drill — it is automation that fires at machine speed. Next, you'll make the autonomy split explicit: which actions may run with no human, which require approval, and why.
