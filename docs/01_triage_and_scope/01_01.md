# Task 01.01 — Open the unified incident in the Defender portal

## Introduction

In Exercise 00 you saw the alert queue the way Acesa's analysts see it — dozens of low-severity items spread across products. Defender XDR has already done something a tired analyst might not: it correlated several of those alerts into **one incident**. Opening that incident is where you start turning noise into a story.

## Description

You will locate the multi-stage incident in the Defender portal, review its attack story and the alerts it spans across identity, email, endpoint, and AI/data, and note how unremarkable each individual signal looks.

## Success Criteria

- You have opened the incident titled **"Multi-stage incident involving Identity, AI, Data, and Infrastructure"** _(illustrative — confirm at build)_
- You have reviewed the alerts it spans across at least four products (identity, email, endpoint, AI/data)
- You can state that each contributing alert is individually low or informational severity

## Key Tasks

### 01: Find the incident in the queue

Locate the single incident that ties the scattered alerts together.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. In the Defender portal, select **Incidents & alerts > Incidents**.

2. Set the time range to the last 24 hours. Find the incident titled **"Multi-stage incident involving Identity, AI, Data, and Infrastructure"** and select it to open the incident page.

3. Note the **Severity**, **Categories** (you should see several tactics — initial access, collection, credential access, lateral movement, execution), and the number of alerts and impacted assets.

   {: .note }
   > 📷 **Screenshot to capture:** the Incidents queue with the multi-stage incident selected, showing its multiple categories and impacted-asset count.

</details>

### 02: Walk the attack story

Read the incident the way it actually unfolded — across layers.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. On the incident page, open the **Attack story** tab and review the incident graph. Trace how the nodes connect a **user**, an **email**, a **device**, a **service principal**, and the **AI application**.

2. Open the **Alerts** tab and review the contributing alerts. You should see signals from multiple detection sources, for example:

   | Detection source | Alert (representative) |
   |------------------|------------------------|
   | Microsoft Entra ID Protection | Impossible travel / atypical sign-in for Mariya Petrova |
   | Defender for Office 365 | Phishing message delivered (look-alike sender) |
   | Defender for Cloud (AI) | Anomalous prompt activity against the Refund Agent |
   | Defender for Identity | Suspicious service-principal / lateral-movement activity |
   | Defender for Endpoint | Suspicious PowerShell execution on OPS-LT-0427 |

3. Notice the **severity** column. Individually, these are the kind of alerts that get auto-closed or buried during a busy shift.

   {: .note }
   > 📷 **Screenshot to capture:** the incident Alerts tab showing the spread of detection sources and their low/medium severities.

</details>

### 03: Preview the assets and entities

Get a first look at who and what is involved.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Open the **Assets** tab and skim the users, devices, and cloud/AI resources the incident touches. Don't catalog them yet — you'll extract entities methodically in Task 01.03.

2. Note anything that already looks load-bearing: the analyst account **Mariya Petrova**, the device **OPS-LT-0427**, and a service principal beginning `sp-refund-agent-…`.

</details>

## Summary

You've opened the one incident that already stitches several products' alerts together and walked its attack story across identity, email, endpoint, and AI/data. You've also seen that every contributing signal is individually unremarkable — the core reason the SOC missed the chain. Next, you'll use Security Copilot to turn this incident into a single, readable cross-domain narrative.
