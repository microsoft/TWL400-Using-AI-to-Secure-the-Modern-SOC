Exercise 02 — Systemic Dependencies Table

Maps each stage-to-stage hop to the trust relationship or design choice that enabled it.

Hop Systemic Dependency That Enabled It Design Change That Breaks It
Stage 1 → 2 The Refund Policy Knowledge Base implicitly trusts authenticated uploads; a valid (stolen) session was sufficient to poison the grounding source — no separate review of grounding-content changes exists. Require grounding-source change review / approval workflow independent of the uploading identity's session validity.
Stage 2 → 3 The Refund Agent trusts its grounding source as authoritative; poisoned content is treated as policy, so prompt injection rides in through RAG rather than being filtered as untrusted external input. Treat RAG-retrieved content as untrusted input; apply content safety filtering at the retrieval layer, not only at the prompt layer.
Stage 3 → 4 sp-refund-agent-inference is over-privileged relative to its task — it holds reach to the Azure OpenAI resource (oai-soclab-v02) in rg-soclab, so a compromised agent context translates directly into infrastructure reach. Scope sp-refund-agent-inference to least-privilege; remove standing access to production AI infrastructure it does not need at rest.
Stage 4 → 5 Endpoint OPS-LT-0427 and the cloud identity share the same compromised user context with no segmentation between cloud and on-device execution planes. Enforce identity segmentation between cloud service-principal context and endpoint user context; device compliance gate before cloud token issuance.
Across all stages Fragmented tooling (50–75 products) is itself a systemic dependency: with no shared correlation layer, each trust boundary is policed in isolation and the full cascade is invisible end to end — no single tool sees the chain. Unified cross-layer correlation (e.g. Microsoft Sentinel + Defender XDR) with analytics rules that join identity, AI, infrastructure, and endpoint signals into one incident.
